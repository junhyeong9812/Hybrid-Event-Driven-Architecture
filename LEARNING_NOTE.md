# 분산 시스템 아키텍처 학습 노트

> 멀티테넌트 | 메시지큐 | Inbox/Outbox 패턴 | 인메모리 이벤트 버스  
> 2026년 2월 25일

---

## 1. 멀티테넌트 (Multi-Tenant)

하나의 소프트웨어 인스턴스나 인프라를 여러 사용자(테넌트)가 공유하는 아키텍처다. 각 테넌트는 독립된 환경처럼 느끼지만, 실제로는 같은 애플리케이션과 인프라를 공유한다. SaaS 서비스(Slack, Salesforce 등)가 대표적인 예시다.

### 1.1 싱글테넌트 vs 멀티테넌트

| 구분 | 싱글테넌트 | 멀티테넌트 |
|------|-----------|-----------|
| 배포 | 고객마다 별도 인스턴스 | 하나의 인스턴스 공유 |
| 비용 | 높음 | 효율적 |
| 유지보수 | 개별 관리 필요 | 일괄 관리 가능 |
| 커스터마이징 | 용이 | 제한적 |

### 1.2 핵심 고려사항

- **데이터 격리**: 테넌트 간 데이터가 절대 섞이지 않아야 한다
- **성능 격리**: 한 테넌트의 과부하가 다른 테넌트에 영향을 주지 않아야 한다 (Noisy Neighbor 문제)
- **확장성**: 테넌트 수 증가에 유연하게 대응해야 한다

### 1.3 데이터 격리 방식

| 방식 | 설명 | 격리 수준 | 비용 |
|------|------|----------|------|
| DB 분리 | 테넌트별 별도 데이터베이스 | 최고 | 높음 |
| 스키마 분리 | 같은 DB, 테넌트별 별도 스키마 | 중간 | 중간 |
| 행 수준 분리 | 같은 테이블에서 `tenant_id`로 구분 | 낮음 | 최저 |

## 2. 메시지큐에서의 멱등성 보장

### 2.1 핵심 문제: 이중 쓰기 (Dual Write)

메시지큐와 DB를 함께 사용할 때 근본적인 문제가 발생한다. "DB 저장 + 메시지 발행"은 서로 다른 시스템에 대한 작업이므로 하나의 트랜잭션으로 묶을 수 없다.

```
1. DB 저장 ✅
2. 메시지 발행 ❌  ← 여기서 실패하면?
→ DB는 변경됐는데 메시지는 안 나감 → 데이터 정합성 깨짐
```

### 2.2 Outbox 패턴 (발행 측)

**목적:** 메시지 발행의 신뢰성 보장 (at-least-once delivery)

DB 트랜잭션 하나로 비즈니스 데이터와 발행할 메시지를 함께 저장한다. 별도 프로세스가 Outbox 테이블을 읽어 메시지큐로 발행하므로, "저장은 됐는데 메시지가 안 나가는" 문제를 원천 차단한다.

```
[하나의 DB 트랜잭션]
1. 비즈니스 데이터 저장
2. Outbox 테이블에 메시지 저장  ← 같은 트랜잭션!

[별도 프로세스]
3. Outbox 테이블을 폴링(또는 CDC)하여 메시지큐로 발행
4. 발행 완료 후 Outbox 레코드 처리 완료 표시
```

### 2.3 Inbox 패턴 (소비 측)

**목적:** 메시지 멱등성 처리 (중복 수신 방지)

at-least-once 전달 방식에서는 메시지가 중복 도착할 수 있다. Inbox 테이블에 message_id를 기록하여 이미 처리한 메시지를 걸러내고, 정확히 한 번만 처리되도록 보장한다.

```
메시지 수신 시:
1. Inbox 테이블에 message_id 존재 확인
2. 이미 있으면 → 스킵 (중복 메시지)
3. 없으면 → Inbox에 message_id 저장 + 비즈니스 로직 처리 (같은 트랜잭션)
```

### 2.4 패턴 요약

| 패턴 | 위치 | 해결하는 문제 |
|------|------|-------------|
| **Outbox** | Producer | 메시지가 반드시 발행됨을 보장 (신뢰성) |
| **Inbox** | Consumer | 중복 메시지를 한 번만 처리 (멱등성) |

> 두 패턴을 조합하면 분산 환경에서 **effectively-exactly-once** 처리를 달성할 수 있다.

## 3. DB의 WAL과 Outbox 패턴 비교

### 3.1 WAL (Write-Ahead Log)

단일 시스템 내에서 crash recovery를 위한 메커니즘이다. 트랜잭션 실행 시 로그를 먼저 기록하고, 이후 실제 반영한다. 크래시가 발생하면 로그를 보고 복구한다.

### 3.2 WAL을 그대로 쓸 수 없는 이유

WAL은 같은 DB 엔진 내부에서 동작한다. 하지만 메시지큐(Kafka, RabbitMQ 등)는 DB와 완전히 별개의 시스템이므로, DB의 WAL이 메시지큐의 발행까지 보장해줄 수 없다. 두 시스템에 걸친 원자적 쓰기가 불가능하다.

### 3.3 개념 비교

| 구분 | WAL | Outbox 패턴 |
|------|-----|------------|
| 동작 | 로그 기록 → 디스크 반영 | 테이블 기록 → 메시지큐 발행 |
| 범위 | 단일 시스템 (DB 내부) | 분산 시스템 (DB + 메시지큐) |
| 목적 | Crash Recovery | 메시지 발행 신뢰성 |

Outbox 패턴은 본질적으로 **애플리케이션 레벨의 WAL**이다.

### 3.4 CDC를 통한 WAL 활용

Debezium 같은 도구가 DB의 트랜잭션 로그(WAL)를 직접 읽어서 Outbox 테이블의 변경 사항을 Kafka로 보내는 방식이다. Outbox 패턴에서 WAL을 실제로 활용하는 가장 가까운 형태이며, 폴링 방식 대비 지연 시간이 짧고 DB 부하가 적다.

## 4. 인메모리 이벤트 버스: 단일 시스템에서의 대안

### 4.1 문제 인식

Outbox/Inbox 패턴은 분산 시스템을 전제로 한다. 모듈러 모놀리스처럼 단일 JVM에서 동작하는 시스템이라면 Kafka 같은 외부 메시지 브로커 없이도 도메인 간 이벤트 통신이 가능하다.

### 4.2 WAL에서 영감을 받은 설계

메모리에 이벤트를 순차적으로 기록(append-only)하고, 각 도메인이 자신의 offset을 기반으로 새 이벤트를 소비하는 구조다. DB의 WAL이 "먼저 기록, 나중에 반영"하는 것처럼, 인메모리 이벤트 스토어도 "먼저 기록, 나중에 처리"한다.

### 4.3 하이브리드 전략

내부 도메인 간에는 인메모리 이벤트 버스를 사용하고, 외부 시스템과의 통신에만 Kafka + Inbox/Outbox를 적용하는 하이브리드 접근이 가능하다. 이렇게 하면 내부는 가볍고 빠르게, 외부는 견고하고 안전하게 동작한다.

| 통신 구간 | 방식 | 일관성 수준 |
|----------|------|-----------|
| 내부 도메인 ↔ 내부 도메인 | 인메모리 이벤트 버스 | 강한 일관성 (같은 트랜잭션) |
| 내부 도메인 → 외부 시스템 | Kafka + Outbox/Inbox | 최종 일관성 (eventually consistent) |

## 핵심 정리

1. **멀티테넌트**는 하나의 인프라를 여러 테넌트가 공유하는 아키텍처로, 데이터 격리와 성능 격리가 핵심이다.
2. **Outbox 패턴**은 Producer 측에서 DB 트랜잭션을 활용해 메시지 발행의 신뢰성을 보장한다.
3. **Inbox 패턴**은 Consumer 측에서 message_id 기반으로 중복 처리를 방지해 멱등성을 보장한다.
4. Outbox 패턴은 개념적으로 WAL과 유사한 **애플리케이션 레벨의 WAL**이다.
5. 단일 시스템에서는 **인메모리 이벤트 버스**로 가볍게 처리하고, 외부 통신에만 Kafka를 사용하는 **하이브리드 전략**이 효과적이다.
6. 이벤트 드리븐 인터페이스를 처음부터 적용하면, 향후 마이크로서비스 전환 시 **통신 계층만 교체**하면 된다.
